## Architecture

This is a **Flutter monorepo** with three layers as sibling top-level directories, composed via `path:` dependencies in `pubspec.yaml`. Each layer is a collection of standalone Dart/Flutter packages, not subfolders of `lib/`.

```
domain/       # Pure models, enums, typedefs. No Flutter/Riverpod imports.
data/         # Repositories: Dio HTTP clients, SharedPreferences wrappers, streams.
application/  # Services: Riverpod providers that expose repositories to the UI layer.
lib/          # Presentation: the nurbanhoney app itself, organized by feature.
```

Dependency rule: `lib/` → `application/*` → `data/*` → `domain/*`. Never invert this. `domain` has no Flutter dependency.

### Package pairs

Most domains come as a **three-package set**: `domain/<name>_domain` (models) + `data/<name>_repository` (I/O) + `application/<name>_service` (Riverpod provider). Examples: `authentication_*`, `dio_*`, `preference_storage*`. When adding a new data source, follow this split rather than dropping code into `lib/`.

### State management (mixed, intentional)

- **Riverpod** (`flutter_riverpod` + `riverpod_annotation`) is the primary pattern. All `application/*_service` packages expose `Provider`/`FutureProvider`/`@riverpod` classes. New features should use Riverpod.
- **Provider + ChangeNotifier** (`lib/common/view/base_view.dart` + `lib/common/viewmodel/base_viewmodel.dart`) is a legacy `BaseView<T extends BaseViewModel>` pattern still present in some feature pages. Don't extend it for new code; migrate toward Riverpod when touching it.

### Entry point

`lib/main.dart` initializes `MobileAds`, `KakaoSdk`, `Firebase`, fetches Android ID, then wraps `App` in a `ProviderScope`. `App` → `AppView` (in `lib/app/view/app.dart`) reads `authenticationServiceProvider` and currently routes both authenticated and unauthenticated users to `HomePageRefactor` — this is intentional work-in-progress, not a bug to "fix" without checking first. The commented-out `HomePage` branch and the `/// 왜 인지 모르겠지만 SplashPage로 안 감` note mark active refactor territory.

### Feature layout in `lib/`

Each feature folder (`home/`, `login/`, `board/`, `article_detail/`, `article_create/`, `myaccount_setting/`, `myaccount_edit/`, `splash/`, `coin/`, `stock/`) follows:

```
<feature>/
  <feature>.dart       # barrel export
  view/
    <feature>_page.dart
    view.dart          # barrel
    widgets/           # feature-local widgets
```

Shared building blocks live in `lib/common/`. Theme (`standardTheme` Provider) comes from `nurbanhoney_ui_service`, backed by colors/text styles in `nurbanhoney_ui_data`.

### Backend

All HTTP calls go through `data/dio_repository` using `Dio()`. The API base URL is hardcoded at `domain/dio_domain/lib/src/models/dio_main_api.dart` → `https://nurbanhoney-api.devjj.co.kr`. Auth token and device ID are stored in `SharedPreferences` via `data/preference_storage`.

### Auth flow

Three login paths converge into a single token stored in `PreferenceStorage`: Firebase Google Sign-In, Kakao SDK, and legacy email/password (`dioRepository.getLogin`). `AuthenticationService` (Riverpod `@riverpod` class) holds `AuthenticationStatus`, and `AuthenticationRepository` exposes it as a stream. On logout, `setUnauthenticated()` clears the token via `PreferenceStorage.setEmptyToken()`.

## Gitignored secrets

The following are required to build but **not** in the repo — obtain them separately and do not commit:

- `lib/config/*` (contains `secret.dart` with `kakaoNativeAppKey` etc. — the entire directory contents are ignored)
- `lib/firebase_options.dart` (generated by `flutterfire configure`)
- `android/app/google-services.json`
- `ios/Runner/GoogleService-Info.plist`
